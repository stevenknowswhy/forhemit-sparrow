## Comparison: Which Version Is Better?

### Version A: Live Search

**Pros:**
- ✅ Real, current pricing data you can actually use
- ✅ Verifiable sources with timestamps
- ✅ Shows what the final output would look like for a real user
- ✅ Demonstrates the agent's value immediately
- ✅ Reveals real-world challenges (blocked sites, pricing variance, UK vs US)

**Cons:**
- ❌ Manual per-product (no automation)
- ❌ No sandbox isolation (I'm searching directly, not in a container)
- ❌ No parallel execution (sequential tool calls)
- ❌ No persistence (results disappear after session)
- ❌ Limited to what web_search + read_webpage can access
- ❌ Cannot do mock checkout, headless browsing, or deep scraping
- ❌ No preference learning or adaptive scheduling
- ❌ No Monte Carlo analysis (would require code execution)

### Version B: Descriptive Spec

**Pros:**
- ✅ Complete technical blueprint for developers
- ✅ Defines exact architecture, data flow, API integrations
- ✅ Specifies edge cases, error handling, confidence filtering
- ✅ Shows the full JSON schema for the app
- ✅ Defines sandbox behavior, proxy rotation, rate limiting
- ✅ Scalable — describes what the production system WILL do

**Cons:**
- ❌ No real data — purely theoretical
- ❌ Doesn't prove the agent can actually find good deals
- ❌ No pricing validation
- ❌ Can't be used by a real user today
- ❌ More useful for engineers than for business decisions

---

## Verdict: Use BOTH

**For the skill, we need a hybrid approach:**

1. **The behavioral prompt** (how I act when invoked) → Version A style
   - Actually search, actually read pages, actually produce a real report
   - This is what the user experiences TODAY

2. **The technical spec** (how the Rust agent will work) → Version B style
   - Defines sandbox, agents, APIs, scoring, edge cases
   - This is what developers reference when building the app

**The skill should contain both:**
- When invoked, it behaves like Version A (live search, real report)
- The report output includes a "Technical Notes" section that maps each step to the corresponding spec element (sandbox, agent, API tier)
- This way the skill is immediately useful AND serves as a living spec that developers can trace back to

**Bottom line:** Version A proves the concept. Version B builds the product. You need both.
